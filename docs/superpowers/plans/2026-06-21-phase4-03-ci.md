**Goal:** Rewrite `.github/workflows/docker-build.yml` to use the owner-namespaced image name,
add `app-v2` as a build/publish trigger, and insert an amd64 smoke-test job that gates the
multi-arch publish. Also touch `ci.yml`: confirm the existing `uv run pytest` step already
collects the sub-plan-01 wheel-contents guard (no new job needed there) and add a
`docker compose config` lint for the sub-plan-02 compose file.

**Architecture:** Two-job Docker workflow: a `smoke` job builds the image for `linux/amd64` with
`--load` (produces a runnable local image), boots the container against a named Docker volume
(inheriting `/config`'s `app:app` ownership from the image), and runs the three contract §I
assertions; a `publish` job gated via `needs: [smoke]` rebuilds for `linux/amd64,linux/arm64`
and pushes to GHCR only on non-PR events. Both jobs share the GHA layer cache so the second
build is incremental. The ci.yml wheel guard is already covered by the existing `uv run pytest`
step, which collects `tests/build/test_wheel_contents.py` automatically.

**Tech Stack:** GitHub Actions (ubuntu-latest); `docker/build-push-action@v6`,
`docker/metadata-action@v5`, `docker/login-action@v3`, `docker/setup-buildx-action@v3`,
`docker/setup-qemu-action@v3`, `actions/checkout@v4` (all at their currently-pinned versions —
do NOT bump without verifying current at execution time per rule 1); Python inside the container
for the migration assertion; `curl` on the runner for the `/health` probe; GHCR
(`ghcr.io`).

## Global Constraints

- **Image name (§A):** `IMAGE_NAME: ${{ github.repository_owner }}/media-scan-monitor` (full
  GHCR ref: `ghcr.io/${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}`). The bare `media-scan-monitor`
  without the owner namespace is an invalid GHCR path.
- **Tag strategy (§A):** `type=ref,event=branch` → `app-v2` tag on `app-v2` pushes;
  `type=raw,value=latest,enable={{is_default_branch}}` → `latest` stays unset until cutover to
  `main`; keep `type=semver` patterns for future `v*` tags.
- **Publish gating (§A):** `push: ${{ github.event_name != 'pull_request' }}` — PRs build +
  smoke but do not push.
- **Port (§B):** `8080` everywhere — the `docker run` port mapping, the health probe URL, and the
  container's internal listen port all read `8080`.
- **Non-root UID (§C):** UID `1000`. The first smoke assertion checks `id -u` returns `1000`.
- **Three smoke assertions (§I), in order:**
  1. `docker exec <c> id -u` → `1000`
  2. `curl -sf http://localhost:8080/health` → HTTP 200 with body containing `"ok"`
  3. Python script (via `docker cp` + `docker exec`) reads `/config/app.db`; uses the installed
     package's `_MIGRATIONS_DIR` + alembic `ScriptDirectory` to compute the expected head
     dynamically; asserts `alembic_version.version_num == expected_head` AND domain tables
     `{server, folder, filetype, setting}` exist.
- **Multi-arch preserved (§A/E):** publish job always builds `linux/amd64,linux/arm64`.
- **No-main-merge context:** this phase (including this workflow change) lands on `app-v2`;
  `latest` stays unset; `main` keeps the legacy `plex_monitor.sh` until the rewrite is a proven
  viable replacement.

## Task 1 — `ci.yml`: confirm the wheel-contents guard + add a `docker compose config` lint

This task has two parts: **(1)** confirm the existing `uv run pytest` step already collects the
wheel-contents guard (no change needed), and **(2)** add a fast `docker compose config` lint so the
compose file authored in sub-plan 02 is validated in CI (spec §"Testing strategy": *"Compose
validation: `docker compose config` as a lint in CI"*). The lint lives in `ci.yml` (the fast
quality workflow, no Docker image build) rather than `docker-build.yml`, so a compose-only edit
doesn't trigger a full image build + smoke + publish.

Sub-plan 01 places the wheel-contents guard at `tests/build/test_wheel_contents.py`. The
question is whether a separate, dedicated ci.yml job is needed or whether the existing `pytest`
step already collects it.

**Analysis:**

1. `ci.yml` path triggers include `tests/**` — adding `tests/build/test_wheel_contents.py` will
   trigger the workflow on every PR or push that changes the test.
2. `uv run pytest` (the current step, line 50 of ci.yml) collects tests recursively by default.
   pytest discovers all `test_*.py` files under the project root, so `tests/build/test_wheel_contents.py`
   is collected with no configuration change.
3. The wheel-contents test calls `uv build --wheel` (or equivalent). `uv` is on PATH in the
   ci.yml runner because `astral-sh/setup-uv@v8.2.0` installs it — no additional setup needed.
4. The spec wanted the wheel check "fast, before the slower smoke test." The smoke test lives
   in `docker-build.yml` (a separate workflow), so there is no ordering constraint within
   `ci.yml`. The wheel test runs on every PR via `ci.yml`'s `quality` job, which finishes in
   under a minute, well before any Docker build starts.

**Conclusion (part 1): the wheel-contents guard needs no new `ci.yml` job.** The existing
`uv run pytest` step already collects it. A dedicated wheel-contents job would duplicate pytest
and add latency.

**Part 2: add the compose lint.** `docker compose` is preinstalled on `ubuntu-latest`, so the
`quality` job can validate the compose file with no extra setup. Add `docker-compose.yml` to the
workflow's path filters (so a compose-only change triggers the lint) and two lint steps to the
`quality` job (default + `init-watches` profile, since the sidecar only materialises under that
profile).

## Task 2 — Rewrite `.github/workflows/docker-build.yml`

**Design choices (documented here so the implementer doesn't re-derive them):**

**Smoke job structure — option B (separate jobs):** The workflow has two jobs: `smoke` (amd64
`--load` + assertions) and `publish` (`needs: [smoke]`, multi-arch push). Option A (one job,
amd64 smoke then multi-arch push as steps) would work but conflates smoke and publish in a single
job, making it harder to see in the GitHub UI whether publish failed vs. smoke failed, and
requiring `if:` guards on the push steps. Option B is cleaner.

**`/config` writability (§D) — named Docker volume:** Contract §D states the image creates
`/config` owned by `app:app`. A new Docker named volume copies the image's directory on first
use, so UID 1000 can write `app.db` and `secret.key` with no runner-side `chown`. This avoids
needing `sudo` in CI. A bind-mount alternative (`mkdir /tmp/msm-config && sudo chown 1000:1000
/tmp/msm-config`) is documented as a comment in the workflow but not used.

**Migration head assertion — dynamic, in-container:** The Python assertion script is written to
a temp file on the runner (`/tmp/assert_migrated.py`), copied into the container via `docker cp`,
and executed via `docker exec`. It imports `mediascanmonitor.db.session._MIGRATIONS_DIR` and
alembic's `ScriptDirectory` from the installed package to compute the expected head at runtime
— no hardcoded revision ID. It also checks that the four domain tables (`server`, `folder`,
`filetype`, `setting`) exist, so a half-migrated DB fails even if the version number matches.

- [ ] **Step 1: Write the complete workflow file**

  Overwrite `.github/workflows/docker-build.yml` with the content below. The file is the
  authoritative deliverable; do not paste code without replacing the file.

```yaml
# Builds, smoke-tests, and publishes the media-scan-monitor Docker image.
#
# Job graph:
#   smoke  → build amd64 (--load) + assert non-root / /health / migrations
#   publish → (needs: smoke) build linux/amd64,linux/arm64 + push to GHCR
#
# On PRs: both jobs run but the publish job does not push (push: false).
# On app-v2 pushes: publishes ghcr.io/<owner>/media-scan-monitor:app-v2.
# On main pushes: also sets the `latest` tag (is_default_branch gate).
# On v* tags: sets semver tags.

name: Build & Smoke-Test Docker Image

on:
  push:
    branches: [main, app-v2]
    tags: ['v*']
    paths:
      - 'mediascanmonitor/**'
      - 'uv.lock'
      - 'pyproject.toml'
      - 'Dockerfile'
      - '.dockerignore'
      - '.github/workflows/docker-build.yml'
  pull_request:
    paths:
      - 'mediascanmonitor/**'
      - 'uv.lock'
      - 'pyproject.toml'
      - 'Dockerfile'
      - '.dockerignore'
      - '.github/workflows/docker-build.yml'
  workflow_dispatch:

env:
  REGISTRY: ghcr.io
  # Decoupled from the repo directory name so the image publishes as
  # ghcr.io/<owner>/media-scan-monitor regardless of what the repo is called.
  # A bare "media-scan-monitor" (no owner) is an invalid GHCR reference.
  IMAGE_NAME: ${{ github.repository_owner }}/media-scan-monitor

jobs:
  # ─────────────────────────────────────────────────────────────────────────────
  # smoke: build amd64-only (--load → runnable local image) and assert:
  #   1. process runs as non-root (UID 1000)
  #   2. GET /health → HTTP 200, body contains "ok"
  #   3. /config/app.db migrated to Alembic head; domain tables present
  #
  # Buildx cannot --load a multi-platform image, so the smoke build is
  # amd64-only. The publish job re-runs the build for both arches (incremental
  # via the shared GHA cache).
  # ─────────────────────────────────────────────────────────────────────────────
  smoke:
    runs-on: ubuntu-latest
    permissions:
      contents: read

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      # No QEMU needed — building only for the runner's native amd64 arch.
      - name: Set up Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build amd64 image (local load for smoke test)
        uses: docker/build-push-action@v6
        with:
          context: .
          platforms: linux/amd64
          load: true
          tags: msm-smoke:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

      # /config writability strategy: use a named Docker volume.
      # When a container mounts a new named volume over a directory that exists
      # in the image, Docker copies the image directory into the volume on first
      # use — inheriting the ownership set by the Dockerfile (app:app, UID 1000).
      # This means UID 1000 can write app.db and secret.key with no runner chown.
      #
      # Alternative (bind mount): mkdir /tmp/msm-config && sudo chown 1000:1000 /tmp/msm-config
      #   docker run ... -v /tmp/msm-config:/config ...
      - name: Start container
        run: |
          docker run -d --name msm-smoke \
            -p 8080:8080 \
            -v msm-smoke-config:/config \
            -e MSM_PASSWORD=smoketest123 \
            msm-smoke:latest

      # Poll /health until the app is up or 60 s elapses.
      # The Dockerfile sets --start-period=20s; allow up to 3× that.
      - name: Wait for /health (up to 60 s)
        run: |
          for i in $(seq 1 30); do
            if curl -sf http://localhost:8080/health > /dev/null 2>&1; then
              echo "Container healthy (attempt ${i})"
              break
            fi
            if [ "${i}" -eq 30 ]; then
              echo "ERROR: container did not become healthy within 60 s"
              docker logs msm-smoke
              exit 1
            fi
            sleep 2
          done

      # ── Assertion 1: non-root ─────────────────────────────────────────────
      # Contract §I·1, §C: the runtime user must be UID 1000 (never root).
      - name: "Assert §I·1 — non-root (UID 1000)"
        run: |
          UID_VAL=$(docker exec msm-smoke id -u)
          echo "Container UID: ${UID_VAL}"
          if [ "${UID_VAL}" != "1000" ]; then
            echo "FAIL: expected UID 1000, got ${UID_VAL}"
            exit 1
          fi

      # ── Assertion 2: /health → 200 with {"status":"ok"} ──────────────────
      # Contract §I·2, §B/§F: port 8080, unauthenticated, body {"status":"ok"}.
      - name: "Assert §I·2 — GET /health returns 200 ok"
        run: |
          RESP=$(curl -sf http://localhost:8080/health)
          echo "Response: ${RESP}"
          if ! echo "${RESP}" | grep -q '"ok"'; then
            echo 'FAIL: expected body to contain "ok"'
            exit 1
          fi

      # ── Assertion 3: migrated to Alembic head ────────────────────────────
      # Contract §I·3: /config/app.db exists and alembic_version.version_num
      # equals the repo's current head. The expected head is computed dynamically
      # from the installed package (mediascanmonitor.db.session._MIGRATIONS_DIR
      # + alembic ScriptDirectory) — never hardcoded. Domain tables are also
      # checked so a half-migrated DB fails even if the version row matches.
      - name: "Assert §I·3 — DB migrated to head, domain tables present"
        run: |
          cat > /tmp/assert_migrated.py << 'PYEOF'
          import sqlite3
          from mediascanmonitor.db.session import _MIGRATIONS_DIR
          from alembic.script import ScriptDirectory
          from alembic.config import Config

          # Compute expected head from the migrations bundled in the installed package.
          cfg = Config()
          cfg.set_main_option("script_location", str(_MIGRATIONS_DIR))
          sd = ScriptDirectory.from_config(cfg)
          heads = [r.revision for r in sd.get_revisions("heads")]
          assert len(heads) == 1, f"expected one head revision, got {heads}"
          expected_head = heads[0]

          # Read what Alembic recorded in the DB.
          conn = sqlite3.connect("/config/app.db")
          ver_row = conn.execute("SELECT version_num FROM alembic_version").fetchone()
          tables = {
              r[0]
              for r in conn.execute(
                  "SELECT name FROM sqlite_master"
                  " WHERE type='table' AND name NOT LIKE 'sqlite_%'"
              )
          }
          conn.close()

          assert ver_row is not None, "alembic_version table is empty — migrations did not run"
          actual = ver_row[0]
          assert actual == expected_head, (
              f"DB not at head: got {actual!r}, want {expected_head!r}"
          )

          required_tables = {"server", "folder", "filetype", "setting"}
          missing = required_tables - tables
          assert not missing, f"missing domain tables: {missing} (found: {sorted(tables)})"

          print(f"OK: at head ({actual}), domain tables present: {sorted(tables)}")
          PYEOF
          docker cp /tmp/assert_migrated.py msm-smoke:/tmp/assert_migrated.py
          docker exec msm-smoke python /tmp/assert_migrated.py

      - name: Dump container logs on failure
        if: failure()
        run: docker logs msm-smoke

      - name: Stop and remove container
        if: always()
        run: |
          docker stop msm-smoke || true
          docker rm -v msm-smoke || true
          docker volume rm msm-smoke-config || true  # named volume; `rm -v` only drops anonymous ones

  # ─────────────────────────────────────────────────────────────────────────────
  # publish: multi-arch (amd64 + arm64) build-and-push, gated on smoke passing.
  #
  # arm64 is preserved for NAS targets (Synology, QNAP).
  # Push is gated on github.event_name != 'pull_request' — PRs run the job but
  # do not write to GHCR.
  #
  # Tag strategy (contract §A):
  #   type=ref,event=branch   → "app-v2" on app-v2 pushes, "main" on main pushes
  #   type=raw,value=latest,enable={{is_default_branch}} → only on main (stays
  #     unset until app-v2 is merged as the viable replacement)
  #   type=semver,pattern=*   → on v* tags
  # ─────────────────────────────────────────────────────────────────────────────
  publish:
    needs: [smoke]
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write   # required to push to ghcr.io

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Buildx
        uses: docker/setup-buildx-action@v3

      # Only log in / push on real branch/tag pushes, not on PRs from forks.
      - name: Log in to GHCR
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata (tags, labels)
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=raw,value=latest,enable={{is_default_branch}}

      - name: Build and push (linux/amd64, linux/arm64)
        uses: docker/build-push-action@v6
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

### What is NOT verifiable locally

- The `publish` job's GHCR push (requires GitHub secrets).
- The `is_default_branch` gate for `latest` (requires a `main` push).
- End-to-end timing of the start-period vs. the 60 s health poll (observable only in a real
  Actions run; the 30-iteration × 2 s loop provides a comfortable margin over the 20 s
  start-period).
